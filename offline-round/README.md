## 내가 탐색한 방향 (최종 미채택)

### 문제 인식
온라인 라운드(W8A8)에서 검증된 접근을 오프라인에서 그대로 밀어붙이기보다, **1.2B급 소형 모델에서 활성화(Activation) 양자화가 성능에 미치는 영향**을 별도로 검증해보고 싶었다. 그래서 W8A8이 아닌 **W8A16**(가중치만 8비트, 활성화는 BF16 유지)으로 방향을 잡았다.

### 시행착오
1. **vLLM 엔진 레벨 커스텀 시도** — 특정 레이어를 BF16으로 강제 고정하려 엔진 소스를 직접 수정 시도. 소형 모델의 잔차 연결(Residual Stream)이 예민해 수치 불안정 발생 + 빌드 시간 20분으로 타임아웃 압박 → 상위 레벨 라이브러리(`llmcompressor`) 레시피 제어로 전략 전환
2. **SmoothQuant 적용 실패** — 활성화 아웃라이어 보정을 시도했으나 EXAONE 아키텍처의 Attention-to-MLP 레이어 매핑을 라이브러리가 지원하지 않아 런타임 에러. Weight-Only(W8A16)로 선회해 안정성 확보
3. **글로벌 Ignore의 함정 (점수 0.61)** — `q_norm`, `k_norm` 등을 레이어 지정 없이 문자열로 통째로 제외했다가 모델 용량이 오히려 폭증, 용량 점수에서 감점
4. **레이어 개수 착각** — 다른 소형 모델 기준으로 23개 레이어라 가정했다가, `print(model)`로 직접 구조를 분석해 EXAONE-1.2B가 실제로는 **32개(0~31) 레이어**임을 확인
5. **경로 매핑 오류** — `embed_tokens`, `layers.0`처럼 축약된 이름을 썼으나 실제 내부 경로는 `model.embed_tokens`, `model.layers.0`이어서 보호 리스트가 무시되고 전부 양자화됨. `named_modules()`로 실제 경로를 직접 추출해 수정

### 최종 접근 (이 폴더의 코드)
- Scheme: **W8A16** (가중치만 8비트, 활성화는 BF16 유지 — 활성화 양자화로 인한 지능 손실을 원천 차단하는 보수적 전략)
- 캘리브레이션: 2048 샘플 / 시퀀스 길이 2048
- Ignore: `lm_head`, `model.embed_tokens`, `model.norm` + 입구 레이어(0, 1) + 출구 레이어(30, 31) 전체를 BF16으로 보존 — 32개 레이어 중 4개만 남겨 용량 증가를 최소화하면서 성능 저하가 가장 큰 구간을 보호

전체 코드는 [`exploration_w8a16.py`](./exploration_w8a16.py) 참고. (원본에 있던 데이콘 API 토큰은 제거함)

### 왜 팀원 버전(0.71)보다 낮았는가 — 추정 원인
코드 로직 자체의 결함이 아니라 아래 요인들의 조합으로 판단된다.

1. **1.2B 모델의 낮은 파라미터 여유(redundancy)** — 7B~13B급 모델은 일부 가중치가 손실돼도 다른 파라미터가 보완하지만, 1.2B는 여유가 거의 없어 Weight-Only 양자화조차 영향이 크게 나타남
2. **캘리브레이션 데이터의 도메인 불일치 가능성** — MANTA-1M(대화형 데이터)로 캘리브레이션했으나 평가셋이 수학/코드/논리추론 등에 치우쳤을 경우 스케일 값이 실제 평가 분포를 충분히 반영하지 못했을 수 있음
3. **평가 산식의 가중치 불균형** — 성능(PerfNorm)을 지키기 위해 4개 레이어 + 임베딩을 BF16으로 남겼는데, 용량(MemNorm)이나 속도(SpeedNorm) 배점이 예상보다 컸다면 그 트레이드오프가 오히려 손해였을 가능성

## 배운 점

- 라이브러리가 최신 아키텍처를 완전히 지원하지 않을 수 있다는 것을 전제하고, 에러 발생 시 우회 전략을 빠르게 설계하는 것이 중요함
- 가정("아마 이럴 것이다")에 의존하지 않고 `print(model)`, `named_modules()` 같은 직접 검증 수단으로 확인하는 습관이 결정적인 버그(레이어 개수 착각, 경로 매핑 오류)를 잡아냄
- 동일 문제에 대해 팀원과 다른 가설(FP8 vs W8A16)로 병렬 실험한 것이 결과적으로 팀 전체의 성공 확률을 높였음
